<objective>
Create `api/app/verification.py`: the single guarded verification state machine that becomes the sole authority over `disc.status` transitions (VERIFY-02). Per D-08 it is a flat-function module (no service class, no FSM library) mirroring the `api/app/disc_identity.py` house style: a one-line docstring, a module-level `LEGAL_TRANSITIONS` frozenset, a domain exception `VerificationTransitionError`, and flat functions taking `db: Session` first.

Per D-09 the general transition table contains ZERO entries targeting `"disputed"`; `disputed` is reachable ONLY through `flag_dispute()`, which refuses to touch an already-`verified` disc â€” closing the silent-flip bug (VERIFY-02 success-criterion #4). Per D-11 the "cannot verify your own submission" rule lives INSIDE `verify()` as a transition invariant. Per Pitfall 4, `verify()` returns a `bool` so route callers preserve their distinct idempotent-200/no-edit vs. auto-verify/edit contracts; do NOT introduce a 400 for `verifiedâ†’verify`.

Purpose: Consolidate the 5 inline `disc.status` mutations scattered across `routes/disc.py` into one grep-able, testable module before Phase 5 raises write concurrency on the same namespace. This module is the Phase 2 seam (D-12).
Output: `api/app/verification.py` + full unit coverage in `api/tests/test_verification.py`.
</objective>
