<objective>
Make the alias check-then-insert write path in `api/app/disc_identity.py` race-safe under concurrent gunicorn workers (IDENT-02, [guardrail]). Per D-01/D-02/D-03: replace the current TOCTOU pattern in `attach_lookup_aliases` (lines 115-128 — `resolve_disc_identity` read, then unconditional `db.add()`) with insert-first / catch `IntegrityError` / re-resolve-the-winner, each insert wrapped in a `db.begin_nested()` SAVEPOINT.

The load-bearing guarantee already exists: `discs.fingerprint` and `disc_identity_aliases.fingerprint` are both `UNIQUE` (migrations `7ffb31fc807f:91`, `900000000003:33`) and re-created for the SQLite test harness. Production runs `gunicorn -w 4` (`docker-compose.prod.yml:34`), so the 4-worker race is real. Per D-02 this is the ONLY option regression-testable on the in-memory SQLite harness (SQLite raises `IntegrityError` on a UNIQUE violation exactly like Postgres); advisory locks / `FOR UPDATE` / SERIALIZABLE are Postgres-only and SQLite silently no-ops them.

Per Pitfall 1, do NOT test via `threading`/`asyncio.gather` (StaticPool serializes — proves nothing). Inject the losing-race state DETERMINISTICALLY and assert convergence, restoring any monkeypatch in `finally` (CLAUDE.md symbol-override rule).

Purpose: A race here would create split/duplicate pressings and corrupt the `dvd1-*` stability guarantee ADR 0001 protects — this MUST land before Phase 5 raises write concurrency.
Output: race-safe `attach_lookup_aliases` (and any shared insert helper) in `disc_identity.py` + deterministic race regression in `api/tests/test_disc_identity_race.py`.
</objective>

| Boundary | Description |
|----------|-------------|
| concurrent workers → shared DB row (fingerprint namespace) | Multiple gunicorn workers may attempt to insert the same disc/alias fingerprint simultaneously; only the DB UNIQUE constraint arbitrates atomically |
| service layer → DB constraint | The service must CATCH the violation and converge, never attempt to prevent the race by reading first (TOCTOU) |
