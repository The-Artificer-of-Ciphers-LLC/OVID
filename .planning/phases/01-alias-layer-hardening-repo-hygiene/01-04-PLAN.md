<objective>
Expose every known identity string for a pressing on the lookup response (IDENT-01): add an additive `fingerprint_aliases` field to `DiscLookupResponse`, sourced from the existing `Disc.identity_aliases` relationship (`models.py:109`). Per D-04 aliases are OBJECTS carrying a method label — `list[{fingerprint: str, method: str, is_primary: bool}]` — via a new `FingerprintAliasResponse` Pydantic model. Per D-05 the top-level `fingerprint: str` stays UNCHANGED (the primary), and the primary is ALSO included in the array flagged `is_primary=true`, so every identity is enumerable from one collection. Per D-06 ordering is primary-first then remaining aliases in INSERTION order — never sorted by string (`dvd1-*`/`dvdread1-*` do not compare meaningfully). Per D-07 the change is strictly additive: `ovid-client`, the ARM shim, and `web/lib/api.ts` keep working; the web interface gains an OPTIONAL field.

Per RESEARCH Pattern 4: `method` is DERIVED from the fingerprint prefix (`fp.split("-", 1)[0]`) via a new `_method_of` serializer helper — `DiscIdentityAlias` has NO `method` column and this phase adds NO migration. Deterministic ordering (D-06) comes from an explicit `order_by` on the relationship (mirroring `Disc.titles`) or a `(created_at, id)` sort in the serializer — pick ONE. `selectinload(Disc.identity_aliases)` is added at all three `_disc_to_response` call sites to avoid N+1.

Purpose: This is the read-side of the alias layer — Phase 7's disc-detail view (WEBUI-02) renders these; the object shape (method + is_primary) survives Phase 5 promotion without a second breaking change.
Output: `FingerprintAliasResponse` + `fingerprint_aliases` in `schemas.py`; `order_by` in `models.py`; `_method_of` + serializer wiring + eager-load in `routes/disc.py`; optional field in `web/lib/api.ts`; extended `test_disc_lookup.py`.
</objective>
