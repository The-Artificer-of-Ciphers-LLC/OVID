<objective>
Wire the five provider callbacks to the pure `resolve_auth` from Plan 04, closing the nOAuth flaw at
the call sites. Refactor `finalize_auth` (routes.py:71) into a thin wrapper that reads session state
(`link_to_user_id`, `web_redirect_uri`, `pending_link_id`), calls `resolve_auth`, and maps the
`AuthResult` to the existing HTTP responses — a 409 merge-offer now carries `pending_link_id`, and the
old session-carried implicit-merge block is removed. Compute each provider's real verified-email
signal at the source: GitHub via `GET /user/emails` (D-05), Google/Apple from their already-verified
id_token claims, Mastodon/IndieAuth unconditionally `email_verified=False` (D-06). Preserve
multi-provider login (AUTH-06) and min-one unlink (AUTH-07).

Purpose: Turn resolve_auth's isolated correctness into live behavior — GitHub/Google sign-in still
work end-to-end (AUTH-01/02) and email merges are confirm-gated (AUTH-08).
Output: refactored routes.py wiring; updated linking + GitHub tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| provider callback → finalize_auth | The verified-email signal must be computed at the provider source, not inferred from a generic profile field |
| session pending_link_id → merge completion | The session value routes the callback; ownership is proven only by the fresh OAuth round-trip |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-08e | Spoofing / Elevation | GitHub email trust | high | mitigate | Email + verified come from GET /user/emails (primary+verified); the unverifiable profile email is display-only (Task 2, Pitfall 2) |
| T-06-08f | Elevation of Privilege | session-carried implicit merge | high | mitigate | The old session pending-link merge block is removed; merge state is a DB PendingAccountLink consumed only by matching re-auth (Task 1, Pitfall 1) |
| T-06-08g | Spoofing | Apple email_verified type confusion | medium | mitigate | String/bool normalization `str(...).lower()=="true"` so "false"/missing are rejected (Task 2, Pitfall 3) |
| T-06-08h | Elevation | Mastodon/IndieAuth email match | high | mitigate | Both pass email_verified=False unconditionally, so resolve_auth never offers a merge for them (Task 2, D-06) |
</threat_model>
