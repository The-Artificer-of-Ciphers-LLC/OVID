<objective>
Build `api/app/auth/merge.py`: the pure, session-free `resolve_auth` function and the
`PendingAccountLink` lifecycle that implement the confirm-gated, re-auth-required account merge
(D-01/D-02/D-04/D-05/D-06/D-07). This is the core nOAuth defense — it decides, at a single testable
choke point, whether a provider login (a) logs into an existing linked account, (b) OFFERS a merge
(verified-email match → PendingAccountLink, no attach), or (c) creates a separate identity, and it
consumes a pending merge ONLY when the same existing account re-authenticates via an already-linked
provider.

This plan is ADDITIVE: it does NOT modify the live `user_upsert`/`finalize_auth` login path. merge.py
is self-contained and unit-tested in isolation, so the existing suite stays green at the Wave 2
boundary. Plan 05 (Wave 3) rewires the provider callbacks to call resolve_auth.

Purpose: Move the trust decision out of session state and out of raw-email equality into a pure,
directly-testable function that satisfies AUTH-08 and AUTH-09's three required cases.
Output: `merge.py` (resolve_auth + AuthResult + exceptions + CRUD helpers); `test_auth_merge.py`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| new-provider email claim → account ownership | An attacker-influenced/unverified email must never be treated as proof of owning an existing account (nOAuth / account pre-hijacking) |
| pending merge state → attach | Attaching a new provider must require re-auth of the existing account, not mere possession of a pending_link_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-08a | Spoofing / Elevation | verified-email merge decision | high | mitigate | A merge is only OFFERED (PendingAccountLink, no attach) and only on a provider-VERIFIED email signal; unverified/attacker emails fork a separate identity (Task 1, D-05/D-07) |
| T-06-08b | Elevation of Privilege | pending-link consume | high | mitigate | Consume requires `str(freshly_authed_user.id) == pending.existing_user_id` proven via an ALREADY-linked provider re-auth; mismatch raises MergeReauthMismatchError (Task 2, D-02) |
| T-06-08c | Tampering / Replay | pending-link reuse | medium | mitigate | Single-use (consumed_at) + TTL (expires_at) enforced at consume; expired/consumed rejected (Task 2) |
| T-06-08d | Elevation | duplicate-email collision on separate identity | medium | mitigate | Separate-identity creation uses a placeholder email when the claimed email collides, so the unique-email constraint blocks silent account overwrite (Task 1, D-07) |
</threat_model>
