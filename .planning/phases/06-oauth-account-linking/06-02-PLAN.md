<objective>
Close the AUTH-05 SSRF gaps in `validate_mastodon_domain`: the current check uses IPv4-only
`socket.gethostbyname`, so an AAAA-only or dual-stack host pointing at a private/loopback IPv6
address bypasses it. Switch to dual-stack `socket.getaddrinfo`, validate every resolved address in
both families (adding `is_reserved`), and lock the two behavioral guardrails AUTH-05 also names —
no redirect-following and no raw-response reflection — with regression tests. Preserve the existing
Mastodon per-instance registration flow (AUTH-04).

DNS-rebinding decision (Claude's Discretion per CONTEXT.md; RESEARCH.md Open Question 1 + A2):
For v0.2.0 this plan scopes AUTH-05 to (a) dual-stack validation at the single validate-before-
registration choke point (`mastodon_login` → `validate_mastodon_domain` → `get_or_register_client`)
and (b) preserving no-redirect / no-reflection. Full IP-pinning custom-transport closure of the
resolve-then-connect TOCTOU time-gap is a DOCUMENTED, ACCEPTED residual risk for v0.2.0 (narrow
window, requires attacker-controlled authoritative DNS + precise timing), recorded in the threat
register below and to be surfaced in docs/auth-setup.md (Plan 07). This is a deliberate, non-silent
scope decision, not an omission.

Purpose: Make the Mastodon instance URL structurally unable to reach internal/reserved addresses.
Output: Hardened `validate_mastodon_domain`; migrated + expanded `test_auth_mastodon.py`.
</objective>

|----------|-------------|
| user-supplied Mastodon instance URL → OVID outbound HTTP | Untrusted hostname controls the target of server-side requests (classic SSRF surface) |
| Mastodon instance response → OVID | Untrusted upstream response could be reflected or redirect-chased into internal targets |
