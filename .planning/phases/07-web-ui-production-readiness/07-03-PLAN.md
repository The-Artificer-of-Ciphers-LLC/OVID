<objective>
Close the two backend findings the D-01 re-review surfaced on the re-integrated multi-disc-set feature:
- **R-1 (anti-echo redaction leak):** `_build_disc_set_nested` (disc.py) and `_build_sibling_summary`
  (set.py) iterate every sibling's `titles` to expose `main_title`, `duration_secs`, and `track_count`
  WITHOUT checking `sibling.status` — leaking derived structural data that a direct
  `GET /v1/disc/{sibling}` would redact for an `unverified` disc (D-09).
- **R-2 (write-ceiling inconsistency):** `POST /v1/set` carries only the volumetric `_dynamic_limit`,
  not the stacked `AUTH_WRITE_LIMIT` per-account ceiling the other three write routes carry (INFRA-04 seam).

Purpose: These are the carried-in re-review items (ROADMAP re-review note; RESEARCH R-1/R-2). R-1 is an
info-disclosure defect on the disc-detail sibling view (WEBUI-02); R-2 is a write-throttle gap on set
creation from the submit flow (WEBUI-03). Both must be fixed before the set/sibling UI counts as
production-ready.
Output: status-gated sibling redaction in both builders + the stacked write ceiling on create_set + pytest.
</objective>

    - `POST /v1/set` (authenticated) is subject to BOTH `@limiter.limit(AUTH_WRITE_LIMIT, methods=["POST"])`

      submit_disc / register_disc behavior.
