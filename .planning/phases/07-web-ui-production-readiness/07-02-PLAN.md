<objective>
Fix the D-04 backend defect: `finalize_auth`'s verified-email merge OFFER currently returns a raw 409
`JSONResponse` (`{error:"email_conflict", pending_link_id}`) that dead-ends the browser on unstyled JSON
on the API host. Change it to a 302 redirect back to the web app's `web_redirect_uri` carrying
`error=email_conflict&pending_link_id=<id>` — mirroring the success-path redirect — while staying
enumeration-safe (ME-02). Also remove the leftover R-3 cruft in `link_provider`.

Purpose: Without this redirect, the settings-page merge banner (07-07, WEBUI-04 / D-05) is unreachable —
the browser never lands back on the web app with the error params it reads. This is the backend half of
the account-linking merge UX.
Output: A redirecting merge-offer branch + explicit link-provider handling + isolated pytest coverage.
</objective>

    Cases (finalize_auth, merge-offer branch — `result.merge_offer is not None`):

    - the 302 `Location` query contains NO `token` and NO `existing_user_id` (ME-02 enumeration guard).
    - session has NO `web_redirect_uri` → returns the existing 409 JSON `{error:"email_conflict", pending_link_id}` (unchanged fallback for API/non-browser callers), still with no `existing_user_id`.
    - `web_redirect_uri` must be popped consistently (it is `.pop`ed once); the merge branch runs before the success-path pop at the old l.193, so read/pop it in the merge branch too.
    R-3 (link_provider): remove the `pass`-only Mastodon/IndieAuth branch + rambling comment; return an explicit 400 `{error:"link_requires_domain"}` (or documented equivalent) for mastodon/indieauth (which need a domain/url a bare POST can't carry), and keep the github/google/apple redirect-to-`/login` behavior. No behavior change for the supported bare-POST providers.
  </behavior>
