<plan_assumption>
UI-SPEC says "dark mode via `@custom-variant dark`", but RESEARCH (Pattern 1, A1, Pitfall) and PATTERNS
resolve this: the app already relies on Tailwind's default `prefers-color-scheme` media strategy for
every `dark:` utility, and no manual theme toggle is in scope. Adding a class/data-attribute
`@custom-variant dark (...)` would OVERRIDE the media strategy and break OS-following dark mode. This
plan therefore KEEPS the media strategy and only adds `@theme` tokens. Documented so it is a decision,
not a silent drop.
</plan_assumption>

| Boundary | Description |
|----------|-------------|
| build-time CSS → browser | Tokens compile to static CSS custom properties; no runtime, no untrusted input crosses here |
| primitive props → DOM | Component props (labels, values) are rendered as JSX text/attributes |

| T-07-01-02 | Information Disclosure | Button/Input/Field | low | mitigate | Primitives render props as JSX text/attributes only; React auto-escapes. No `dangerouslySetInnerHTML`. |
