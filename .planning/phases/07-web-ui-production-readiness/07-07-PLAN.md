<task type="checkpoint:decision" gate="blocking">
  <name>Task 1: Decide the add-provider browser-flow mechanism (research Open Q1)</name>
  <decision>How does the authenticated "Link a provider" flow start an OAuth round-trip that the backend can attribute to the current user?</decision>
  <context>
    `POST /v1/auth/link/{provider}` (verified in api/app/auth/routes.py l.864-878) requires Bearer auth via
    `get_current_user`, sets `link_to_user_id` in the SESSION cookie, then 302s to `/v1/auth/{provider}/login`.
    The web app stores its JWT in localStorage, not a cookie. A top-level browser navigation can't send the
    Bearer header; a Bearer `fetch` can't carry the browser through the cross-origin provider redirect. In
    production the web (oviddb.org) and API (api.oviddb.org) are different origins. This is research Open Q1 —
    it is NOT delegated to Claude's discretion in CONTEXT and has potential backend scope, so it needs a decision.
  </context>
  <options>
    <option id="option-b">
      <name>Frontend-only: prime the session cookie via credentialed fetch, then top-level navigate (RECOMMENDED)</name>
      <pros>No backend change; reuses existing `POST /link` + `/login` endpoints; stays within D-05 "middle depth"; keeps this plan single-subsystem (web only).</pros>
      <cons>Requires credentialed CORS for the web origin and a SameSite=Lax session cookie that survives a top-level GET navigation across the api/web subdomains; depends on infra allowing credentialed CORS (verify during 07-08 staging wiring).</cons>
    </option>
    <option id="option-a">
      <name>Backend signed-token: add a signed short-lived link_to_user_id param to /login</name>
      <pros>Robust across the api/web subdomain split; a plain top-level navigation starts an authenticated link; no cookie/CORS dependency.</pros>
      <cons>Auth-security-sensitive backend change to the per-provider /login endpoints in api/app/auth/routes.py (Phase 6 territory); larger scope; would make this plan cross-subsystem (add api/app/auth/routes.py to files_modified).</cons>
    </option>
    <option id="option-c">
      <name>Plain provider-login buttons (no link_to_user_id) — REJECTED</name>
      <pros>Trivial.</pros>
      <cons>Does NOT link a new provider: a fresh login with a new email creates a SEPARATE account, and a matching email triggers the email_conflict merge instead of a link. Incorrect for "add provider" semantics. Documented as rejected, not selectable.</cons>
    </option>
  </options>
  <files>(decision only — no files modified; the choice is recorded in 07-07-SUMMARY.md and consumed by Tasks 2-3)</files>
  <action>Present the three options above to the developer and record the selected add-provider browser-flow mechanism (default option-b, frontend-only). If option-a is selected, note that api/app/auth/routes.py joins this plan's changes (wave-safe, since this plan runs after 07-02).</action>
  <verify>Decision (option-a or option-b) recorded before Task 3 begins.</verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (localStorage JWT) → API auth/link | Authenticated add/remove requests cross here |
| API redirect (?error) → settings page | Untrusted query params drive the merge banner |
| session cookie ↔ OAuth round-trip | link_to_user_id must survive provider redirect (add-provider flow) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-07-01 | Information Disclosure | merge banner (ME-02) | high | mitigate | Banner names only current-account providers; renders no email/user-id/existing_user_id and never states a different-account match. Test asserts absence. |
| T-07-07-02 | Elevation of Privilege | nOAuth silent merge (AUTH-08) | high | mitigate | UI only OFFERS + links to re-auth (GET /login?pending_link_id); no client path auto-completes a merge; the DB PendingAccountLink + re-auth (resolve_auth) remains the sole completion path. |
| T-07-07-03 | Tampering | add-provider flow / open redirect | high | mitigate | Link/login navigation targets go through `getBaseUrl()` + the API's CORS_ORIGINS-allowlisted `_validate_web_redirect_uri` (fail-closed); no token is placed in any redirect the client constructs. |
| T-07-07-04 | Access Control | unlink last method | medium | mitigate | Server enforces `cannot_unlink_last` (400); UI mirrors by disabling the last control; both layers present. |
| T-07-07-05 | Tampering | XSS via provider strings in banner | low | mitigate | Provider names rendered as JSX text; React auto-escapes; no `dangerouslySetInnerHTML`. |
</threat_model>
