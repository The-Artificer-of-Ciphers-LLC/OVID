<objective>
Prove **API p95 ≤ 500ms against the actual Redis-backed, multi-worker gunicorn configuration** (INFRA-03) with a Locust harness, an extended bulk seed, and a non-blocking scheduled CI job. Per D-00 this plan runs LAST — after the Redis backbone (Plan 01), the write throttle (Plan 02), and the deployment wiring (Plan 03) land — so the load test validates the real target config, never the retiring `memory://` one (D-14).

Purpose: A p95 claim is only meaningful against the honest stack (Postgres, `gunicorn -w 4`, live Redis-backed limiter). The gate is kept off the per-PR critical path (D-12) because an absolute-ms assertion on GitHub's noisy shared runners would produce intermittent failures this project must root-cause every time.

Output: a bulk-capable `api/scripts/seed.py` (+ a unit test), a `loadtest/` Locust harness with a native exit-code p95 gate, and `.github/workflows/loadtest.yml` (workflow_dispatch + schedule, non-blocking, artifact-publishing).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner → ephemeral postgres/redis services | Load-test services must be reachable by the job but never by untrusted networks; must carry no real secrets or production data |
| load harness → API auth | The submit task needs an authenticated token; leaking a real/long-lived credential into the harness or logs would be a disclosure |
| seed tooling → database | The seed writes test-only rows; must not touch or mirror production data |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-08 | Information Disclosure | load-test/seed tooling shipping secrets or opening DB/Redis to untrusted networks in CI | medium | mitigate | GH Actions `services` are bound to the runner only (no public exposure), ephemeral per-run; the submit token is minted at runtime for a SEEDED test user (never a real credential) and read from an env var, not committed; seed inserts synthetic test rows only |
| T-03-09 | Tampering | pip install of `locust` in the load-test job | medium | mitigate | Package Legitimacy Audit complete (RESEARCH — `locust` verdict OK, first-party `locustio/locust`); pinned `locust>=2.44,<3.0` in an isolated `loadtest/requirements.txt`; not an API runtime dep; no `[ASSUMED]`/`[SUS]` package, so no blocking-human checkpoint |
| T-03-10 | Denial of Service (self-inflicted) | submit task tripping the Plan 02 write cap and reporting false failures | low | accept | Handled in the harness (429 marked non-failure / unique fingerprints / spread tokens); a benign measurement artifact, not a service risk |
</threat_model>
