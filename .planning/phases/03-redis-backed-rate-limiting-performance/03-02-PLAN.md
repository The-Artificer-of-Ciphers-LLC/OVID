<objective>
Add a **tighter, method-scoped per-account write ceiling** on the three write routes by stacking a second `@limiter.limit(AUTH_WRITE_LIMIT, methods=["POST"])` decorator above the existing `@limiter.limit(_dynamic_limit)` — delivering INFRA-04 basic abuse prevention live in production. This closes the D-09 gap: today a flood of brand-new fingerprints (never confirmations) gets no write ceiling because the Phase-2 cooldown only fires on the confirmation branch.

Purpose: Reads are read-heavy and public (untouched at 100/500 per minute); writes are human-paced disc submissions that deserve a coarse volumetric cap keyed per account. The cap is meaningful only now that Plan 01 made Redis the shared cross-worker store (D-00) — under `memory://` a 20/min cap inflates ~4× across workers.

Output: three stacked write decorators, a one-line note distinguishing this coarse request-rate ceiling from the Phase-2 semantic confirmation cooldown (D-10 seam), and a test proving both the 429 and the no-double-count layering.
</objective>
